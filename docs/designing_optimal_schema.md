On **database / data model schema**, an “optimal schema” is not the cleanest-looking schema. It is the schema that best fits the system’s **truth, workload, change patterns, and failure modes**.

A good schema answers:

> What entities exist, what facts must remain true, what queries must be fast, what changes often, and what mistakes must be impossible?

## What an optimal schema looks like

Usually, it has these traits:

### 1. Clear domain entities

Tables or collections map to real concepts, not random UI screens.

Bad:

```text
dashboard_data
user_stuff
resume_json
analysis_result
```

Better:

```text
users
documents
document_versions
jobs
job_requirements
candidate_profiles
resume_versions
agent_runs
agent_steps
validation_findings
```

The schema should make the system understandable without reading all the application code.

---

### 2. Separation between source data, derived data, and events

This is especially important in LLM/data apps.

You generally want to separate:

```text
Raw input
→ Parsed/structured representation
→ Derived analysis
→ User-facing output
→ Logs/events/evaluations
```

Example for a resume optimizer:

```text
documents
document_versions
parsed_resume_sections
job_postings
job_requirements
candidate_profile_facts
agent_runs
agent_steps
agent_outputs
validation_findings
resume_drafts
resume_exports
```

The key idea: don’t overwrite truth with model interpretation.

Bad:

```text
resume {
  raw_text,
  improved_text,
  score,
  reasoning,
  job_url,
  model_output,
  final_docx_path
}
```

Everything is mashed together.

Better:

```text
documents
- id
- user_id
- type
- storage_url
- created_at

document_versions
- id
- document_id
- raw_text
- parsed_json
- parser_model
- created_at

agent_runs
- id
- user_id
- job_posting_id
- resume_document_id
- status
- started_at
- completed_at

agent_steps
- id
- run_id
- step_name
- model
- input_hash
- output_json
- latency_ms
- token_count
- cost_usd
- created_at

validation_findings
- id
- run_id
- claim
- evidence_source
- verdict
- confidence
- explanation
```

This makes debugging, replaying, auditing, and improving the system much easier.

---

### 3. Normalized where correctness matters, denormalized where speed matters

A common beginner mistake is thinking “optimal schema” means fully normalized.

Not always.

For transactional truth, normalize:

```text
users
orders
payments
documents
permissions
```

For analytics or UI speed, denormalize:

```text
run_summary_view
daily_usage_metrics
document_search_index
```

A good system often has both:

```text
Core normalized tables
+
Materialized views / summary tables
+
Vector/search indexes
+
Event logs
```

So the criterion is not “3NF everywhere.” The criterion is:

> Can the schema preserve truth while serving the real queries efficiently?

---

### 4. Stable IDs and versioning

For any serious app, especially LLM apps, versioning is non-negotiable.

You want:

```text
id
created_at
updated_at
version
source_id
run_id
model_name
prompt_version
schema_version
```

Because outputs are not timeless. A resume analysis generated by GPT-5.5 with prompt v3 is not the same artifact as one generated by Claude or Gemini with prompt v7.

For LLM/data pipelines, every important derived object should be traceable back to:

```text
input data
model
prompt
parameters
pipeline version
timestamp
user action
```

That is what makes the schema evaluable instead of just usable.

---

### 5. Constraints make invalid states impossible

Good schema design pushes correctness into the database where possible.

Examples:

```sql
user_id NOT NULL
status CHECK (status IN ('pending', 'running', 'failed', 'completed'))
created_at DEFAULT now()
UNIQUE(user_id, document_id, version)
FOREIGN KEY (run_id) REFERENCES agent_runs(id)
```

Bad schema lets impossible states exist:

```text
agent_run.status = "done"
completed_at = null
output_id = null
error_message = "timeout"
```

Good schema makes the state machine explicit:

```text
agent_runs
- status: pending | running | failed | completed | cancelled
- started_at nullable
- completed_at nullable
- error_code nullable
- error_message nullable
```

And ideally, the app enforces valid transitions:

```text
pending → running → completed
pending → running → failed
running → cancelled
```

## Criteria for an optimal schema

I’d use these criteria.

### 1. Correctness

Can it represent the domain accurately?

Ask:

```text
Can one user have multiple documents?
Can one document have multiple versions?
Can one run use multiple models?
Can one output be regenerated?
Can two users share the same source document?
Can a validation finding cite multiple evidence sources?
```

If the schema cannot express real-world cases, it is not optimal.

---

### 2. Query fitness

Design from real queries, not abstract beauty.

Example queries:

```text
Show all runs for this user.
Show the latest resume version.
Find failed agent steps.
Compare model cost by pipeline stage.
Retrieve chunks relevant to this job requirement.
Show all claims that failed validation.
```

A schema is good when important queries are natural.

Bad sign:

```text
To answer a basic product question, you need to parse giant JSON blobs everywhere.
```

JSON columns are fine, but they should not replace core relational structure.

---

### 3. Evolvability

The schema should survive product changes.

Bad:

```text
resume_score
```

Better:

```text
scores
- id
- object_type
- object_id
- rubric_id
- score_name
- score_value
- judge_model
- created_at
```

This lets you add multiple scores later:

```text
ats_score
evidence_score
specificity_score
seniority_alignment_score
risk_score
```

without constantly altering the core table.

---

### 4. Traceability

For LLM systems, this is huge.

You should be able to answer:

```text
Why did the system produce this output?
Which input caused it?
Which model generated it?
Which prompt version?
Which retrieval chunks were used?
Which validation checks passed or failed?
How much did it cost?
```

A schema without traceability makes your app feel like magic, but impossible to debug.

---

### 5. Performance

Optimal schema matches expected load.

You need indexes on common access paths:

```sql
CREATE INDEX idx_agent_runs_user_created
ON agent_runs(user_id, created_at DESC);

CREATE INDEX idx_agent_steps_run
ON agent_steps(run_id);

CREATE INDEX idx_documents_user_type
ON documents(user_id, type);
```

For vector/RAG systems:

```text
chunks
- id
- document_version_id
- text
- chunk_index
- metadata
- embedding
```

And indexes for:

```text
document_version_id
user_id
embedding vector index
full-text search
created_at
```

Performance is not just about speed. It is also about avoiding future rewrites.

---

### 6. Minimal redundancy, intentional duplication

Redundancy is bad when accidental, good when intentional.

Bad redundancy:

```text
user_email copied into 12 tables
job_title copied everywhere
resume_text copied into every agent step
```

Good redundancy:

```text
agent_steps.input_snapshot
agent_steps.output_snapshot
run_summaries.total_cost
run_summaries.final_score
```

Because these are audit snapshots or read-optimized summaries.

The rule:

> Duplicate for history, auditing, or speed. Don’t duplicate because the model is unclear.

---

### 7. Clear ownership and lifecycle

Every entity should have an owner and lifecycle.

Example:

```text
documents belong to users
document_versions belong to documents
chunks belong to document_versions
agent_steps belong to agent_runs
validation_findings belong to agent_runs
exports belong to resume_drafts
```

Also ask:

```text
What gets deleted when a user deletes their account?
What is retained for audit?
What can be regenerated?
What is immutable?
What expires?
```

This matters a lot for privacy and storage cost.

---

## My practical schema-quality checklist

A schema is strong if you can say yes to most of these:

```text
1. Core entities have clear names.
2. IDs are stable and not overloaded.
3. Important relationships are explicit.
4. Raw data and derived data are separated.
5. LLM outputs are versioned and traceable.
6. Common queries are simple.
7. JSON is used for flexible details, not core truth.
8. Constraints prevent invalid states.
9. Indexes match query patterns.
10. Historical artifacts are not silently overwritten.
11. The schema can support analytics without painful joins everywhere.
12. The schema can evolve without massive migrations every week.
```

## For your kind of LLM/data apps, the “optimal” pattern is probably this

```text
users
projects

documents
document_versions
document_chunks
document_embeddings

external_sources
source_snapshots

jobs
job_requirements

candidate_profiles
candidate_profile_facts
evidence_items

agent_runs
agent_steps
agent_messages
agent_artifacts

retrieval_queries
retrieval_results

validation_checks
validation_findings

scores
rubrics
evaluations

exports
usage_events
cost_events
```

That may look like a lot, but conceptually it is clean:

```text
Inputs
→ Structured facts
→ Retrieval
→ Agent execution
→ Validation
→ Evaluation
→ Output
→ Usage/cost analytics
```

The deeper point: for modern LLM apps, the schema should not only store app data. It should store **epistemic provenance**.

Meaning:

> What did the system know, where did it come from, what did the model infer, what evidence supported it, and what version of the pipeline produced it?

That is what separates a toy LLM app from a serious data-engineered LLM product.
